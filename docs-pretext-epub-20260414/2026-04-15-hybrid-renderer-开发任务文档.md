# 基于 Pretext 的混合渲染开发任务文档

## 1. 文档目标

本文档用于将《基于 Pretext 的混合渲染需求文档》拆分为可执行开发任务，作为当前仓库实现“章节级 `canvas/dom` fallback”的顺序与验收依据。

约束原则：

- 所有任务合计必须覆盖混合渲染需求文档中的“本轮实现需求”
- 本文档只拆分章节级 fallback，不拆分块级 fallback
- 每个任务必须带测试要求
- 开发顺序默认从上到下执行
- 未完成前置任务，不进入后置任务

关联文档：

- [混合渲染需求文档](./2026-04-15-hybrid-renderer-requirements.md)
- [Canvas 渲染需求文档](./2026-04-15-canvas-renderer-requirements.md)
- [Reflowable EPUB 兼容增强需求文档](./2026-04-15-reflowable-compat-requirements.md)
- [现有开发任务文档](./开发任务文档.md)

## 2. 总体执行规则

### 2.1 任务执行规则

- 每个任务完成后，必须先补测试，再进入下一个任务
- 每个任务必须提交可运行代码，不接受只有设计没有实现的完成态
- 每个任务必须同步检查对 `packages/core` 与 `packages/demo` 的影响
- 混合渲染引入后，不允许破坏当前 `canvas` 主渲染链路
- DOM fallback 必须通过适配层接入，不允许把正文主状态重新交给浏览器 DOM 直接主导

### 2.2 测试执行规则

- 每个任务至少包含单元测试
- 涉及章节判断与渲染分流的任务必须补集成测试
- 涉及阅读行为的任务必须补 demo 验证步骤或 e2e
- 涉及 fallback 规则的任务必须补“命中条件 / 未命中条件”双向测试
- 涉及状态同步的任务必须补 TOC、搜索、跳转和进度相关回归测试

### 2.3 完成定义

一个任务仅在以下条件都满足时才算完成：

- 代码完成
- 测试完成并通过
- 类型检查通过
- 相关 demo 或文档更新
- 不破坏当前 `scroll / paginated / search / toc / hitTest` 核心能力

## 3. 当前执行范围说明

本轮只实现章节级 fallback：

- 章节预处理与归一化
- 章节复杂度分析器
- 章节级 `canvas/dom` 渲染模式决策
- DOM 章节受控渲染路径
- 统一阅读状态层适配

本轮不实现：

- 块级 fallback
- 同章 `canvas + dom` 混排
- DOM block 测高与跨页
- 统一选区与批注模型

## 4. 任务分阶段拆分

## 阶段 A：章节级渲染决策基础设施

### A1. 定义章节渲染模式与决策结果类型

目标：

- 在 `packages/core` 中定义 `RenderMode`、`ChapterRenderDecision` 等基础类型
- 明确章节级渲染分流所需的最小数据结构
- 为后续 analyzer、runtime 和 demo 共用决策结果打基础

输出：

- 章节渲染模式类型
- 章节决策结果类型
- 决策结果导出入口

测试要求：

- 类型测试验证公开类型导出稳定
- 单元测试验证默认决策结果结构与字段完整性

### A2. 建立章节分析输入模型

目标：

- 为 analyzer 建立统一输入模型
- 把章节 DOM、节点计数、标签分布、样式特征等分析数据集中到同一层
- 明确分析器只消费预处理后的章节数据，不直接深入 runtime 逻辑

输出：

- analyzer input builder
- 章节分析输入类型

测试要求：

- 单元测试覆盖基础标签统计、节点数统计、嵌套深度统计
- 单元测试验证分析输入在空章节、普通章节、复杂章节上的稳定输出

## 阶段 B：章节预处理与归一化

### B1. 建立章节预处理入口

目标：

- 在章节进入渲染前建立统一预处理入口
- 归一化 DOM 结构、标签信息和样式输入
- 过滤无意义空节点并保留正文语义

输出：

- chapter preprocess entry
- 归一化章节结果类型

测试要求：

- 单元测试覆盖空白节点过滤与文本保留
- 单元测试覆盖常见章节结构归一化结果

### B2. 统一 Canvas 与 DOM 路径的章节输入

目标：

- 让 `canvas` 章节和 `dom` 章节尽可能复用同一份预处理章节结果
- 避免两条路径各自重复构建章节结构
- 为后续章节模式切换和调试提供统一输入基线

输出：

- 共享章节输入适配层

测试要求：

- 集成测试验证同一章节预处理结果可以分别进入 `canvas` 与 `dom` 路径
- 回归测试验证现有 `canvas` 章节主链路不被破坏

## 阶段 C：复杂度分析器

### C1. 实现高风险标签与复杂样式识别

目标：

- 识别 `table`、`svg`、`math`、`iframe` 等高风险标签
- 识别 `float`、`position`、`flex`、`grid` 等高风险样式
- 输出命中原因列表

输出：

- high-risk tag detector
- complex-style detector

测试要求：

- 单元测试覆盖每类高风险标签与样式的命中判断
- 单元测试覆盖未命中时不误判

### C2. 实现章节复杂度评分

目标：

- 建立可维护的评分规则
- 输出 `score` 与 `reasons`
- 支持基于阈值的 `canvas/dom` 判定

输出：

- chapter complexity scorer
- render mode decision function

测试要求：

- 单元测试覆盖低复杂度章节判定为 `canvas`
- 单元测试覆盖高复杂度章节判定为 `dom`
- 单元测试覆盖多因素叠加下的阈值判断

### C3. 实现章节决策缓存

目标：

- 基于章节内容、样式输入和配置生成稳定缓存键
- 避免相同章节重复分析
- 保证配置变化后可正确失效

输出：

- chapter render decision cache

测试要求：

- 单元测试覆盖缓存命中与缓存失效
- 集成测试验证主题、字号、容器变化不误伤纯结构分析缓存

## 阶段 D：Reader 章节分流能力

### D1. 在 runtime 中接入章节级渲染分流

目标：

- 在章节进入渲染前执行 analyzer
- 根据 `RenderMode` 分流到 `canvas` 章节路径或 `dom` 章节路径
- 保持对外 Reader API 不变

输出：

- runtime 分流入口
- render mode 到章节 renderer 的路由逻辑

测试要求：

- 集成测试覆盖同一本书不同章节走不同路径
- 回归测试验证纯 `canvas` 书籍仍可正常打开与渲染

### D2. 暴露章节渲染模式的调试与诊断信息

目标：

- 为调试和 demo 提供当前章节渲染模式、复杂度评分和命中原因
- 保证调试信息不侵入正式阅读逻辑

输出：

- render diagnostics API 或调试只读字段

测试要求：

- 单元测试验证诊断结果与 analyzer 输出一致
- demo 验证步骤确认可以观察当前章节模式

## 阶段 E：DOM 章节渲染路径

### E1. 建立受控 DOM 章节渲染器

目标：

- 为 `dom` 章节建立独立 renderer
- 受控渲染章节正文，不重新接管阅读器整体 UI
- 支持章节切换、销毁与重建

输出：

- dom chapter renderer
- dom chapter mount / unmount 逻辑

测试要求：

- 集成测试验证 `dom` 章节可正确挂载与卸载
- 回归测试验证切换 `canvas` 与 `dom` 章节时不会残留旧节点

### E2. 完成 DOM 章节样式归一化

目标：

- 统一字体栈、字号、行高、主题样式
- 对复杂但非必要的花哨样式做归一化
- 降低 DOM 章节与 `canvas` 章节的视觉割裂

输出：

- dom style normalization layer

测试要求：

- 单元测试覆盖字体、字号、行高和主题样式映射
- 集成测试覆盖复杂章节进入 DOM 后的基础视觉约束

### E3. 建立 DOM 章节基础交互能力

目标：

- 支持基础链接点击
- 支持章节内基础点击定位
- 支持最小可用的复制与文本选择保留

输出：

- dom chapter interaction adapter

测试要求：

- 集成测试覆盖链接跳转与章节定位
- demo 验证步骤覆盖 DOM 章节基础交互

## 阶段 F：统一阅读状态层对齐

### F1. 对齐章节切换、目录跳转与 href 跳转

目标：

- 保证 `canvas` 与 `dom` 章节都能接入统一章节切换逻辑
- TOC 和 `href` 跳转不依赖具体渲染模式
- 保持 `goToHref` 和目录跳转对外行为一致

输出：

- 统一章节定位适配层

测试要求：

- 集成测试覆盖 `canvas -> dom`、`dom -> canvas` 的跨章节跳转
- 回归测试覆盖 TOC 跳转

### F2. 对齐搜索结果与命中跳转

目标：

- 搜索继续基于章节文本模型而不是渲染层实现
- 搜索命中能够定位到 `canvas` 或 `dom` 章节
- 保持搜索面板和阅读区跳转一致

输出：

- search to chapter locator adapter

测试要求：

- 集成测试覆盖搜索命中落在两种渲染模式章节中的跳转
- 回归测试验证命中高亮与当前章节同步

### F3. 对齐阅读进度与当前章节状态

目标：

- 保证两种章节路径共用同一套进度计算语义
- 当前章节、滚动位置和分页位置对外语义稳定
- 章节模式切换不导致进度跳变异常

输出：

- progress synchronization adapter

测试要求：

- 集成测试覆盖滚动模式与分页模式的章节进度同步
- 回归测试验证模式切换后进度结果稳定

## 阶段 G：Demo 与回归体系

### G1. 为 demo 增加章节模式观测能力

目标：

- 在 demo 中可观察当前章节走 `canvas` 还是 `dom`
- 可查看 analyzer 的基础决策信息
- 支持最小调试入口，便于验收

输出：

- demo 调试面板或章节模式标识

测试要求：

- demo 验证步骤覆盖 `canvas` 章节和 `dom` 章节两种显示

### G2. 建立章节级 fallback fixture

目标：

- 新增至少一组专门用于章节级 fallback 的样本
- 样本中同时包含应走 `canvas` 的章节与应走 `dom` 的章节
- 为 analyzer、runtime 和 demo 回归提供固定输入

输出：

- fallback fixture
- fixture 说明文档

测试要求：

- 单元测试或集成测试读取 fixture 并验证章节决策结果
- 文档说明覆盖每个 fixture 命中的规则点

### G3. 建立章节级 fallback 的端到端回归

目标：

- 通过 demo smoke 或 e2e 验证混合渲染在浏览器中可运行
- 覆盖章节渲染模式切换、TOC 跳转、搜索跳转等关键路径

输出：

- e2e 回归用例

测试要求：

- 端到端测试至少覆盖一个 `canvas` 章节和一个 `dom` 章节
- 端到端测试覆盖一次章节模式切换后的阅读行为

## 5. 本轮不拆分任务的后续需求

以下内容属于需求文档中已经记录、但本轮不进入任务拆分的后续阶段：

- 块级 fallback
- 同章 `canvas + dom` block 混排
- DOM fallback block 测高与分页
- DOM block 整块挪页或整章降级策略
- 统一片段级命中、选区、批注模型

这些内容在本轮任务完成后，再根据章节级 fallback 的运行结果单独立项。

## 6. 验收顺序建议

建议执行顺序如下：

1. `A1 -> A2`
2. `B1 -> B2`
3. `C1 -> C2 -> C3`
4. `D1 -> D2`
5. `E1 -> E2 -> E3`
6. `F1 -> F2 -> F3`
7. `G1 -> G2 -> G3`

建议验收顺序如下：

1. analyzer 能稳定输出章节决策结果
2. 复杂章节可整章降级为 `dom`
3. 简单章节继续稳定走 `canvas`
4. TOC、搜索、跳转、进度在两种章节模式下均可用
5. demo 中可实际观察章节级 fallback 生效
