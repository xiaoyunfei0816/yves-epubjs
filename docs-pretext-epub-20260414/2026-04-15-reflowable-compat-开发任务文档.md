# 基于 Pretext 的 Reflowable EPUB 兼容增强开发任务文档

## 1. 文档目标

本文档用于将《基于 Pretext 的 Reflowable EPUB 兼容增强需求文档》拆分为可执行开发任务，作为当前仓库提升常见 `reflowable EPUB` 文本书兼容率的实现顺序与验收依据。

约束原则：

- 所有任务合计必须覆盖兼容增强需求文档的全部范围
- 开发任务以当前仓库现状为起点，不重复规划已完成的容器解析和 canvas 主链路
- 每个任务必须带测试要求
- 开发顺序默认从上到下执行
- 未完成前置任务，不进入后置任务

关联文档：

- [Reflowable EPUB 兼容增强需求文档](./2026-04-15-reflowable-compat-requirements.md)
- [Canvas 渲染需求文档](./2026-04-15-canvas-renderer-requirements.md)
- [现有开发任务文档](./开发任务文档.md)

## 2. 总体执行规则

### 2.1 任务执行规则

- 每个任务完成后，必须先补测试，再进入下一个任务
- 每个任务必须提交可运行代码，不接受只有设计没有实现的完成态
- 每个任务必须同步检查 `packages/core` 与 `packages/demo` 的行为影响
- 引入第三方库后，必须在代码中建立清晰的适配层，避免业务层直接依赖底层库细节
- 兼容增强过程中，不允许破坏当前 EPUB 主打开链路、分页主链路和 canvas 主渲染链路

### 2.2 测试执行规则

- 每个任务至少包含单元测试
- 涉及解析链路串联的任务必须补集成测试
- 涉及用户可见阅读行为的任务必须补 e2e 或 demo 验证步骤
- 涉及兼容降级的任务必须补“未知标签/未知样式”测试
- 涉及样本书的任务必须沉淀为可重复回归的 fixture

### 2.3 完成定义

一个任务仅在以下条件都满足时才算完成：

- 代码完成
- 测试完成并通过
- 类型检查通过
- 相关文档或样本更新
- 不破坏当前 `scroll / paginated / search / toc / hitTest` 核心能力

## 3. 任务分阶段拆分

## 阶段 A：第三方解析基础设施接入

### A1. 引入 HTML/CSS 解析依赖并建立适配层

目标：

- 引入 `htmlparser2`、`domhandler`、`css-select`、`css-tree`
- 在 `packages/core` 中建立项目内部适配层，隔离第三方库 API
- 明确“章节 XHTML 解析”和“CSS 解析/选择器匹配”的统一入口

输出：

- 依赖更新
- HTML DOM 适配模块
- CSS AST 适配模块
- selector matching 适配模块

测试要求：

- 单元测试验证 HTML 解析适配层可输出稳定节点结构
- 单元测试验证 CSS 解析适配层可正确解析基础规则
- 类型检查确保业务层不直接依赖第三方库内部类型

### A2. 保留现有 XML 主链路并完成职责边界拆分

目标：

- 保持 `container.xml / OPF / NAV / NCX` 仍走现有 XML 解析器
- 明确章节 XHTML 与书籍元信息解析的边界
- 防止引入新依赖后误改现有 OPF/NAV/NCX 解析路径

输出：

- parser 层职责说明
- 章节解析入口重组

测试要求：

- 集成测试验证 EPUB 打开主链路不回归
- 单元测试验证章节内容解析器仅作用于 XHTML spine 文档

## 阶段 B：章节 DOM 中间层

### B1. 将章节 XHTML 解析升级为 DOM 中间树

目标：

- 使用新 HTML 解析链路替代当前章节内容对象化解析方式
- 为节点保留 `tagName`、属性、父子关系和兄弟顺序
- 统一处理 XHTML 中的文本节点、元素节点和空节点

输出：

- DOM 中间树构建器
- DOM 节点访问工具

测试要求：

- 单元测试覆盖标题、段落、列表、表格、figure 等典型 DOM 结构
- 快照测试验证节点顺序、属性和文本节点保留正确

### B2. 建立 DOM 到内容模型的转换骨架

目标：

- 在 DOM 中间层之上建立“块级节点识别”和“行内节点识别”骨架
- 保留未知节点的统一降级入口
- 为后续样式计算与结构映射提供可扩展接口

输出：

- DOM walker
- block/inline 转换骨架

测试要求：

- 单元测试覆盖已知节点识别
- 单元测试覆盖未知节点递归降级而不丢正文内容

## 阶段 C：CSS 解析与样式白名单

### C1. 解析章节关联 CSS 资源

目标：

- 从 manifest 和章节引用关系中提取样式资源
- 解析每个章节关联的 CSS 文本
- 建立 CSS AST 缓存

输出：

- CSS 资源读取器
- CSS AST cache

测试要求：

- 单元测试覆盖单个与多个样式表解析
- 集成测试验证章节可正确找到关联样式资源

### C2. 实现基础选择器匹配

目标：

- 支持元素、类、ID、后代选择器的节点匹配
- 建立基础 specificity 处理逻辑
- 为节点生成有序规则集合

输出：

- selector matcher
- 规则优先级排序逻辑

测试要求：

- 单元测试覆盖元素、类、ID、后代选择器匹配
- 单元测试覆盖规则优先级与覆盖顺序

### C3. 实现 CSS 白名单和计算样式合并

目标：

- 从 CSS 声明中提取白名单属性
- 合并默认样式、外链样式和内联样式
- 为块级节点和行内节点生成计算样式

输出：

- style whitelist processor
- computed style resolver

测试要求：

- 单元测试覆盖允许样式进入计算结果
- 单元测试覆盖未知声明被忽略
- 集成测试验证 class + inline style 合并结果正确

## 阶段 D：内容模型扩展

### D1. 扩展统一内容模型类型

目标：

- 扩展 `BlockNode`、`InlineNode` 与相关类型
- 为节点增加 `tagName`、`className`、`style`、`lang`、`dir` 等通用属性
- 支持 `figure`、`figcaption`、`dl/dt/dd`、`aside`、`nav`、`sub/sup`、`small`、`mark`

输出：

- 类型定义更新
- 内容模型升级说明

测试要求：

- 类型测试验证公开类型可正常导出
- 单元测试验证新节点结构可构造、可访问

### D2. 实现块级结构映射增强

目标：

- 让 DOM 到块级模型转换支持 `figure`、`figcaption`、`dl/dt/dd`、`aside`、`caption`
- 让列表、表格和图文结构具备更完整的结构化输出
- 未识别块级节点默认递归解析其子内容

输出：

- 增强版 block parser

测试要求：

- 单元测试覆盖新增块级标签映射
- 快照测试验证复杂章节结构输出

### D3. 实现行内结构映射增强

目标：

- 支持 `span`、`br`、`sub`、`sup`、`small`、`mark`、`del`、`ins`、inline `img`
- 未识别行内标签按 `span` 降级
- 保证嵌套 inline 结构顺序稳定

输出：

- 增强版 inline parser

测试要求：

- 单元测试覆盖新增行内标签
- 快照测试验证复杂嵌套 inline 输出

### D4. 保持锚点、脚注和内部链接可用

目标：

- 让扩展后的内容模型仍然稳定维护 anchor 索引
- 支持常见脚注引用和尾注引用结构
- 保证链接 href、锚点和 block 绑定关系不回归

输出：

- anchor/index builder 增强版

测试要求：

- 单元测试覆盖同章与跨章锚点
- 集成测试覆盖脚注/尾注跳转结构

## 阶段 E：布局层兼容增强

### E1. 让 text-like block 支持更多 inline 能力

目标：

- 让 `LayoutEngine` 支持 `br`
- 支持 `sub/sup` 的基础排版偏移
- 支持 `small`、`mark` 等文本样式
- 避免因为常见 inline 节点存在就整体退回粗糙 native 布局

输出：

- `LayoutEngine` text-like block 增强

测试要求：

- 单元测试覆盖换行、上下标、小号文本
- 集成测试验证常见富文本段落仍走 pretext 主路径

### E2. 实现 `pre` 和代码块保真布局

目标：

- 支持 `white-space: pre-wrap` 或等价保真策略
- 让代码块在保留内容可读性的前提下稳定布局
- 处理长行与换行策略

输出：

- `pre/code` 布局增强

测试要求：

- 单元测试覆盖缩进、换行、空格保留
- 集成测试验证代码块分页和滚动可用

### E3. 实现列表、图文和表格的基础布局模型

目标：

- 支持列表 marker、缩进和嵌套层级
- 支持 `figure + figcaption` 的上下结构
- 支持表格列宽、单元格边框和基础 padding

输出：

- list layout
- figure layout
- table layout v1

测试要求：

- 单元测试覆盖嵌套列表与 marker 计算
- 单元测试覆盖 figure 高度与图注布局
- 集成测试覆盖表格布局结果稳定

## 阶段 F：Canvas 渲染兼容增强

### F1. 实现新增 block/inline 的绘制能力

目标：

- 为 `mark`、`sub/sup`、`figure`、`figcaption`、`aside`、增强版 list/table 提供可读绘制
- 保证新增计算样式能进入 draw op
- 保持命中区域和最终绘制边界一致

输出：

- `DisplayListBuilder` 增强
- `CanvasRenderer` painter 增强

测试要求：

- 单元测试覆盖新增 draw op 生成
- 视觉快照或集成测试验证新增结构可读

### F2. 优化图片和 inline image 行为

目标：

- 支持 inline image 的行内占位
- 支持 `figure` 图片和普通图片的差异化绘制
- 保持图片异步加载后的局部刷新能力

输出：

- image draw path 增强

测试要求：

- 单元测试覆盖 inline/block image 差异
- 集成测试验证图片晚到时正文和定位不回退

## 阶段 G：运行时行为与文本提取增强

### G1. 更新搜索、文本提取和命中测试

目标：

- 让 `reader` 对新 block/inline 类型的文本提取完整可用
- 保证搜索索引不遗漏 `figure`、表格、定义列表和图注文本
- 让命中测试继续适配新增结构

输出：

- `reader` 文本提取逻辑增强
- hit test 补强

测试要求：

- 单元测试覆盖新增节点文本提取
- 集成测试验证搜索和跳转在兼容样本下可用

### G2. 维持 locator、分页与滚动稳定

目标：

- 让兼容增强后的结构仍可正确参与分页和滚动
- 避免结构复杂度提升导致 locator 丢失或页码漂移
- 保持 `scroll / paginated` 两种模式行为一致

输出：

- 运行时定位与分页补强

测试要求：

- 集成测试覆盖复杂章节下的翻页、滚动与跳转
- e2e 测试验证目录、脚注和搜索结果定位

## 阶段 H：样本书与回归体系

### H1. 建立兼容样本集

目标：

- 为常见 reflowable 文本书建立 fixture 和最小章节样本
- 覆盖小说、教材、图文、脚注、嵌套列表和表格场景
- 明确哪些样本用于 parser，哪些用于 renderer，哪些用于 e2e

输出：

- `test-fixtures` 新增兼容样本
- 样本说明文档

测试要求：

- 新增样本可被测试代码稳定读取
- 每个样本至少对应一个解析或渲染测试

### H2. 建立兼容回归测试矩阵

目标：

- 形成 parser、style resolver、layout、renderer、reader 的回归矩阵
- 为未知标签和未知样式建立显式回归
- 建立常见 EPUB 样本的 smoke 回归

输出：

- 测试矩阵说明
- 新增单测、集成测试和 e2e

测试要求：

- `pnpm test` 可跑通核心兼容回归
- `pnpm test:e2e` 覆盖 demo 中的关键阅读路径

## 阶段 I：文档与验收

### I1. 输出兼容性边界说明

目标：

- 明确支持标签、支持样式和降级策略
- 明确本阶段不支持内容
- 给宿主和后续开发者提供可读边界

输出：

- 兼容性说明文档
- README 或技术文档补充

测试要求：

- 文档链接正确
- 示例与当前实现一致

### I2. 完成最终验收清单

目标：

- 根据需求文档逐项验收
- 记录真实通过项、收敛项和遗留问题
- 为下一阶段扩展保留清晰边界

输出：

- 验收清单
- 遗留问题列表

测试要求：

- 本地完成 `typecheck`、`test`、`build`
- 关键样本在 demo 中人工回归通过

## 4. 建议执行顺序

建议按以下顺序推进：

1. `A1 -> A2`
2. `B1 -> B2`
3. `C1 -> C2 -> C3`
4. `D1 -> D2 -> D3 -> D4`
5. `E1 -> E2 -> E3`
6. `F1 -> F2`
7. `G1 -> G2`
8. `H1 -> H2`
9. `I1 -> I2`

原因：

- 先把第三方库和 DOM/CSS 基础设施接好，后面的内容模型和样式层才有稳定输入
- 内容模型扩展必须先于布局与渲染增强
- 布局与渲染增强完成后，再补运行时和回归体系，返工成本更低

## 5. 首批实现建议

如果按最小可交付批次推进，建议第一批只做以下任务：

- `A1`
- `A2`
- `B1`
- `B2`
- `C1`
- `C2`
- `C3`
- `D1`
- `D2`
- `D3`

第一批完成后，项目应达到：

- 已引入成熟第三方 HTML/CSS 解析能力
- 章节内容不再依赖当前窄白名单对象解析
- 常见未知标签开始进入可控降级路径
- 后续布局和 renderer 增强具备稳定输入基础
